Brief · One pager

The page is the music stand.

HarmonyLab is one person's lead-sheet workshop. The redesign treats the song like a document being studied, not a SaaS dashboard. Notation goes on top. Chords become editable cells. Roman numerals look like Roman numerals. Stub features get out of the way. The user's key-center colors stay user data.

Promoting an inferred chord

Single click → edit popover opens. The "Accept as written" button writes the inferred symbol to Chords.chord_symbol and clears the is_inferred flag → next render shows it as composer-written.

PUT /chords/{id} · sets chord_symbol + is_manual_edit=true · is_inferred=false
Why three cues, not one?

A jazz pianist working from a printed score must never mistake a machine guess for what the composer wrote. Italics alone reads as a stylistic flourish; opacity alone reads as muted. Stacked cues survive printing, screenshots, and quick glances.

Why does Jobim use G♭7 here instead of a more conventional ii–V?
claude
G♭7 in m.6 is the tritone substitute of C7, which would normally function as V/ii (resolving to Fm or F7). Jobim collapses the conventional Gm7 — C7 — F7 into Gm7 — G♭7 — F7♭9 so the bass walks chromatically down a half-step at every measure (G → G♭ → F).

Spec 01 · highest-stakes interaction

Editing a chord symbol inline.

Single click into the cell. Type. ⏎ commits, Esc cancels. Adjacent fields (Roman, function, voicing, comment) live in a small popover anchored below the cell — open by default when the cell goes into edit, but the user can dismiss with Esc and still keep typing the chord symbol.

step 1
Idle

Chord cell shows symbol + roman + function. Cursor hovering on cell raises bg one step.

no requests
step 2 · click
Enter edit

Cell gains amber outline. Symbol becomes editable in place; cursor placed after last char. Popover with related fields slides up below cell.

no requests · local state
step 3 · type
Live preview

Each keystroke is validated against /vocabulary/chord-symbols (debounced 240ms). Invalid symbols render in rose; valid keep ink-0. Quickpicks show jazz aliases.

GET /api/v1/vocabulary/chord-symbols (cached 1h)
step 4 · ⏎
Commit

Two writes fire in parallel: PUT /chords/{id} updates the chord row; PUT /analysis/songs/{id}/chord/{idx} records the override. Optimistic UI — cell goes "edited" state immediately.

PUT /chords/482 + PUT /analysis/songs/12/chord/6
step 5 · toast
Confirm

Bottom toast appears: "Saved · m.7 → F7♭9". 200ms underline flash on the cell. ⌘Z within 8s reverts via DELETE override. After 8s, undo only via explicit edit.

DELETE /analysis/songs/12/chord/6 (undo path)

Failure modes

  • Invalid symbol — commit refused, cell stays in edit, rose error tooltip below input. No request issued.
  • Network error — toast becomes rose: "Couldn't save · retrying in 3s". Cell shows pending badge until retry succeeds or fails terminally.
  • Conflict (override exists) — PUT merges fields. No conflict modal; the override row is the source of truth.
  • Chord_index drift — depends on the FK migration noted in Part 4 of the brief. Until migrated, edits use chord_id and the index is recomputed server-side.
Spec 02

Re-analyzing with a manual key override.

When automatic key detection is wrong (low confidence, or the user is studying the song in a different key), they tap the pencil next to the key pill, pick a new key, and the entire analysis recomputes against it.

step 1
Pencil ✎ on key pill

Inline popover appears with all 24 keys arranged in circle of fifths. Current detected key shown faintly. A "use detected" option resets to algorithm value.

no request
step 2 · pick
Set manual override

POST /analysis/songs/{id}/manual-key with {key: "F minor"}. SongAnalysis.manual_key_override is written. UI shows "key · F min (manual)" pill in amber.

POST /analysis/songs/12/manual-key
step 3 · reanalyze
Re-analyze runs

POST /analysis/songs/{id} recomputes Roman numerals, function labels, and KeyRegions against the new tonic. Skeleton shimmer ONLY here (genuinely slow, ~600ms-2s).

POST /analysis/songs/12
step 4 · result
Grid + timeline refresh

All chord cells re-render with new RNs. Key-center timeline rebuilds. Old overrides are preserved (they're per-index user data, not algo output).

GET /analysis/songs/12 (refetch)

Note: Manual key override is per-song, persists in SongAnalysis.manual_key_override. There is no "session" concept — the user's pick stays until they clear it via use detected.

Spec 03

Importing a PDF + resolving parser errors.

OMR is the lossy pipeline. The interaction must let the user fix bad reads BEFORE the song hits the library — preview-then-commit, not import-then-pray.

step 1
Drop PDF

Drag onto Import Drawer → OMR tab. File is hashed (sha256), checked against existing song_imports rows for duplicates.

local — hash + dup check
step 2

Rationale · one page

Why this, not that.

Five design decisions, justified against the brief and inventory.

1 · Notation pane goes on top

The brief says "the notation pane should never feel like a sidebar widget — it should feel like the document you're studying." That rules out the obvious fourth-tab implementation. Notation lives in a fixed top pane on the Chords tab (collapsible but expanded by default), and gets the entire viewport on the Notation tab. Chord cells become the editable analytical layer beneath the score, not a competing visual surface.

Trade-off: above-the-fold real estate cost. Mitigated by aggressive collapse-to-strip (40px tall) for users in pure-edit mode.

2 · Edit pattern: click-cell + popover, drawer for depth

Modal is wrong (interrupts flow). Right-click context menu is wrong (no chord context preserved). Permanent sidebar is wrong (too many fields, low density on a dense grid). Inline cell edit + popover for related fields is the only pattern where the user keeps their eye on the chord they're editing. Drawer slides in only when they invoke deeper work (AI debate, override history, voicing presets).

Keyboard: ⏎ commit, Esc cancel, ⇥ to next field in popover, ⌘↩ commit + advance to next cell.

3 · Inferred chords get three cues

Italics alone reads as a stylistic flourish. Opacity alone reads as muted. A dotted underline alone reads as a link. Stacked — italic + dotted underline + 60% opacity + leading · + "inferred" in the function slot — survives black-and-white printing, low-contrast displays, and accidental color-blindness in any one channel.

Acceptance: single click → "Accept as written" promotes inferred to composer-written; one PUT sets is_manual_edit = true, clears the inferred flag.

4 · Stub features go to /lab

Improvisation, Quiz, Progress, and RLHF all have 0 rows of real data after months of availability. They cost top-level cognitive real estate the user can't afford. All four move to /lab — a single tertiary nav item — preserving the code paths until PL decides to revive, redesign, or deprecate.

Recommendation: keep all four available in /lab — none should be deprecated. Low usage to date reflects AI context limitations (a work-in-progress in HarmonyLab and across the field) rather than wrong-fit features. See §stub features verdict.

5 · The user's colors are data, not theme

UserPreferences.key_center_colors is the only user-scoped surface in the app that matters daily. The redesign treats it as core data: chord cells get a 3px left-strip in the user's color for that key; key pills inherit it; the analysis timeline is colored by it. The brand accent (amber) is reserved for selection, primary actions, and section labels — never key identity.

6 · No multi-user UI, anywhere

Per brief Part 4. The user-chip in the topbar is decorative (logout hook) — no avatar grid, no "shared with," no "@mention" anywhere. The single auth surface is /settings (which already requires JWT). Every other route stays open by design.

New BE required · summary

What CAI needs to add for HM44.

Aggregated from the matrix above. Six discrete changes — none are large. Ordered by dependency.

  1. FK migration · ChordAnalysisOverrides.chord_id (REPLACES chord_index) Schema · highest priority · per brief Part 4. Blocks inline editing.
  2. Disable BUG-007 fill-forward in analysis service Logic · blocks notation pane from rendering wrong chords. Must precede OSMD integration.
  3. Add Chords.is_inferred BIT column + populate from inference algo (TBD) Schema + logic · enables inferred-chord visual treatment.
  4. Add Chords.voicing_notation NVARCHAR(50) Schema · brief §3 REQ-2. PUT /chords/{id} body extended.
  5. Surface Songs.raw_xml in GET /songs/{id} (or new /xml sub-route) API shape change · OSMD must receive MusicXML. Verify raw_xml is captured by all import paths.
  6. Add MusicXML export endpoint: GET /exports/musicxml/{song_id} Brief §3 REQ-4. Mirror existing MuseScore export pattern.
  7. Expand GET /songs/ response with fs_modified_at (REQ-017) + capability flags Join to song_imports for fs_modified_at; expose has_raw_xml boolean.
  8. KeyRegions CRUD endpoints (optional, P2) ~4 routes for drag-to-override key boundaries. Defer if HM44 is tight.

Free wins (no new BE, just new FE callers)

  • Export menu — wires existing GET /exports/musescore/{id} into the topbar of song detail.
  • Chord-symbol autocomplete in the edit popover — wires GET /vocabulary/chord-symbols (cached) into the live preview validator.
  • Per-chord CRUD on edit popover — uses PUT/DELETE /chords/{id} (already there).
  • Chord comments — uses Chords.comments (column exists, no FE before today).
  • WebMIDI status pill in topbar — uses GET /midi/webmidi-check.
